# Pharmacophore Report Generator

## Purpose

Combine a set of per-PDB pharmacophore JSON files (typically produced by the `pharmacophore-analyzer` skill) into a single multi-structure analysis report aimed at virtual-screening teams. The report identifies orthosteric vs non-orthosteric ligands by interaction-filter retention, surfaces the conserved receptor residues that anchor each pharmacophore feature, derives a consensus query (mandatory + optional features) when one isn't already provided, and presents the result in a consistent format that can be handed to a screening tool (Pharmer, align-it, RDKit shape screening, MOE, LigandScout).

The skill exists because the per-PDB JSONs already contain rich data — partner residue, atom name, distance, interaction type — that almost never gets surfaced in downstream documentation. The job here is to turn that latent data into a screening-ready writeup with a residue-resolved interaction map, retention statistics, and a defensible query.

## When to Use

This skill triggers when the user has:

- Run `pharmacophore-analyzer` on **multiple PDB structures** of the same receptor (typically 3–20) and wants a unified report.
- A directory containing per-PDB `*.pharmacophore.json` files (with `_filtered` and optionally `_all` variants), and asks to "summarize", "write up", "compare across structures", or "build a consensus".
- An existing pharmacophore project that needs reorganization and a refreshed analysis report.

Do **not** use this skill for:

- Single-structure pharmacophore extraction → use `pharmacophore-analyzer`.
- Running the actual virtual screen → that's a downstream task with a different toolchain.
- Receptors with only one cocrystal → there's nothing to consensus-cluster; a per-structure pharmacophore description is sufficient.

## Expected Input Layout

The skill expects something like this (the exact filenames may vary, but the per-PDB grouping with `_filtered`/`_all` suffixes is the canonical convention — generated by `pharmacophore-analyzer` and maintained downstream):

```
<project>/
├── raw/{pdb_id}/{pdb_id}_clean.pdb              CXCR4-style monomer + ligand PDB files (needed for Cα alignment)
├── pharmacophore/                                 (or any name)
│   ├── consensus_pharmacophore.json               OPTIONAL — if absent, the skill derives one
│   └── {pdb_id}/
│       ├── {pdb_id}{_chain}_filtered.pharmacophore.json   REQUIRED
│       ├── {pdb_id}{_chain}_filtered.pharmacophore.pml    optional
│       ├── {pdb_id}{_chain}_all.pharmacophore.json        optional but recommended (drives retention stats)
│       └── {pdb_id}_clean.pdb                              symlink to raw/
└── <receptor>_holostructures.csv                  metadata index — pdb_id, ligand_comp_ids, resolution, method
```

If the user's layout differs (e.g., `results/filtered/`, flat dir, etc.), the skill should still work — the bundled script accepts a `--per-pdb-dir` argument that auto-detects the structure.

The skill also needs to know the **receptor name and UniProt accession** — ask the user if not obvious from the directory or the metadata CSV.

## Workflow

The skill is a four-stage pipeline. Each stage produces a concrete artifact that the next stage consumes.

### Stage 1 — Run `extract_report_data.py` (deterministic data extraction)

Always run the bundled script first. It does the tedious arithmetic and emits a single structured JSON that drives the entire report. The script lives in `scripts/extract_report_data.py` of this skill — invoke it via the skill's resolved path:

```bash
python <skill-dir>/scripts/extract_report_data.py \
  --project-dir <project_path> \
  --receptor-name "Receptor Name" \
  --uniprot-id "PXXXXX" \
  --output <project_path>/.report_data.json
```

Replace `<skill-dir>` with the directory this SKILL.md lives in (the skill harness resolves this). Only requires Python 3 and numpy; no other dependencies.

What the script computes (so Claude doesn't have to re-derive):
- **Per-ligand retention**: copies, mean retention, all-feature totals → drives the orthosteric/excluded classification
- **Residue engagement**: for every filtered feature in every PDB, the partner residue/atom/distance/interaction type, aggregated by residue across PDBs (with conservation flags)
- **Family retention**: per-family raw vs filtered counts across the orthosteric set → drives the §5.5 retention table and the "include/exclude" guidance
- **File manifest**: every output file with a one-line role annotation
- **Numbering offsets**: detects when the same conserved residue appears under different PDB-specific numbering (e.g., ASP262 in 8U4P vs ASP270 in 8ZPL/8ZPN — an 8-residue construct offset) and flags it

If `--derive-consensus` is also passed (default: only if no `consensus_pharmacophore.json` is present), the script additionally:
- Picks a reference PDB (the orthosteric structure with the most filtered features)
- Kabsch-aligns Cα coordinates of all other orthosteric clean PDBs to the reference
- Transforms feature coordinates into the aligned frame
- Greedy single-linkage clusters features by family within `2 × tolerance`
- Classifies clusters as **mandatory** (sources ≥ ⌈N/2⌉) or **optional** (sources < ⌈N/2⌉)
- Writes `consensus_pharmacophore.json` next to the per-PDB outputs

See `references/consensus_derivation.md` for the math and `scripts/extract_report_data.py --help` for all options.

### Stage 2 — Review and decide

After reading `.report_data.json`, take a moment to:

- **Confirm the orthosteric/excluded split.** The script auto-classifies based on retention threshold (default 25%), but borderline ligands deserve a sanity check — open the per-PDB filtered JSON and inspect. A ligand with 10% retention may still be orthosteric if it's only present in one structure with poor density.
- **Spot the conserved residues.** Identify the top 1–3 residues that absorb most of the engagement and note any numbering aliases (the script flags these in `numbering_offsets`).
- **Identify receptor-class context.** GPCR? Use TM6/TM7/ECL2 descriptors. Kinase? Use hinge / DFG / αC. Nuclear receptor? Use H12 / AF-2. Use whatever vocabulary fits the receptor — but be honest: don't invent context you can't verify from the residue numbering alone.

### Stage 3 — Compose the report

Write `<project_path>/pharmacophore/REPORT.md` (or wherever the user's report lives) following the canonical template in `references/report_template.md`. The template has 10 sections plus an Executive Summary stub at the top. Don't deviate from this structure unless the user asks — consistency across receptors is the point of having a template.

The most distinctive (and most often skipped) sections are §4.6 Receptor Residue Engagement and §4.7 Global Interaction Map. These are where the partner-residue data from `.report_data.json` becomes prose — populate them aggressively.

### Stage 4 — Verify and report back

Before declaring done:
- Open the report and check that all numbers in tables (counts, retention percentages, residue contact totals) come from `.report_data.json`, not Claude's mental arithmetic.
- Confirm `consensus_pharmacophore.json` exists and has the expected feature count (matches the report's §5.1 query summary).
- Tell the user what was produced and where, plus any caveats the script flagged (low-quality SMILES, numbering offsets, low-retention ligands that needed a manual call).

The full template lives in `references/report_template.md` — read it before writing the report. Headline structure:

```

## Executive Summary
## 1. Dataset and Preprocessing
## 2. Pharmacophore Feature Detection
## 3. Ligand Classification: Orthosteric vs Non-Orthosteric
## 4. Orthosteric Pharmacophore Derivation

   ### 4.3 Aligned Pharmacophore Similarity
   ### 4.4 Per-Feature Match

   ### 5.1 Query Summary
```

## Tone and Style

Match the tone defined in `references/report_template.md` (the bundled canonical template — read it before writing). Specifically:

- **Scientific but action-oriented.** Each section has a concrete output (a table, a query, a recommendation), not just narrative.
- **Tables over prose** wherever the data is tabular — feature lists, interaction summaries, retention stats, screening strategies.
- **Caveats up front, not buried.** §7 Data Quality is mandatory; bond-order failures, numbering offsets, low-source-count consensus features all belong there.
- **No hedging in the headline finding.** The Executive Summary should commit to a specific claim: "the pocket presents a Donor + PosIonizable–dominated pharmacophore anchored on residue X (TM6) and residue Y (TM7)" — not "the analysis suggests that some hydrogen bonds may be present."
- **Cross-references to file paths** so the reader can verify any claim by opening the underlying JSON or PML.

## Adapting Across Receptor Classes

The template is receptor-agnostic but Claude should adapt the **biological context** based on what kind of receptor this is:

- **GPCRs** (CXCR4, ATR-style chemokine, β2AR, etc.) — describe pocket residues by TM helix (TM2, TM6, TM7) and extracellular loop (ECL2, ECL3); Ballesteros-Weinstein numbering (Asp^6.58, Glu^7.39) if known.
- **Kinases** (ATR, ABL, EGFR) — use hinge region, DFG motif, αC helix, P-loop; categorize ligands by ATP-competitive vs allosteric (Type I/II/III).
- **Nuclear receptors** — H3/H5/H12, AF-2 surface, ligand-binding pocket residues.
- **Enzymes (proteases, esterases)** — catalytic triad members, oxyanion hole, S1/S2 pockets.

If you're not sure what class the receptor is, ask the user — don't guess. The pocket descriptors are too important to get wrong.

## Reference Files

- `references/report_template.md` — full canonical template with table formats, prose conventions, and renumbering rules. Read before writing.
- `references/consensus_derivation.md` — math + algorithm for the Kabsch alignment + greedy clustering used in Stage 1's `--derive-consensus`.
- `scripts/extract_report_data.py` — the bundled data extraction + consensus derivation tool. `--help` lists all options.

## Limitations

- The Kabsch alignment assumes the receptor structures are the same protein — cross-receptor consensus is out of scope.
- Greedy clustering is order-dependent for borderline cases; for receptors with many close-spaced features, the clustering threshold (`2 × tolerance` by default) may need adjustment.
- The skill cannot validate prospective screening performance — the report's §6 covers internal consistency only. Any "this query will retrieve X% of actives" claim requires a separate retrospective benchmark.
- Receptor numbering offsets between PDB constructs are detected but **not normalized**; the report flags them as a caveat. UniProt-canonical renumbering would require a separate sequence alignment step.
